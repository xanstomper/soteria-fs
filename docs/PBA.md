# Pre-Boot Authentication (PBA) Design

## What is PBA?

A Pre-Boot Authentication (PBA) is a small program that runs **before
the operating system** and prompts the user for credentials. After
successful authentication, it unlocks the OS volume and chains to the
OS bootloader.

PBA is what allows full-disk encryption to work: without it, the OS
would need to read an unencrypted bootloader, which an attacker
could replace. With PBA, the entire boot chain is authenticated.

## Soteria PBA Architecture

```
┌─────────────────────────────────────────────┐
│  UEFI Firmware                               │
│  - Verifies Secure Boot signatures           │
│  - Loads the Soteria PBA binary from ESP     │
│    at \EFI\Soteria\bootx64.efi               │
└─────────────────────────────────────────────┘
                  ↓
┌─────────────────────────────────────────────┐
│  Soteria PBA (soteria-pba)                   │
│  1. Display banner (legally required)        │
│  2. Prompt for passphrase (if configured)    │
│  3. Derive volume key via Argon2id           │
│  4. Decrypt the OS volume header             │
│  5. Verify XTS-key-check                    │
│  6. (if TPM) Extend PCR 5 with new state     │
│  7. (if TPM) Unseal the master key           │
│  8. Pass the master key to the OS driver     │
│  9. Chain-load the OS bootloader             │
└─────────────────────────────────────────────┘
                  ↓
┌─────────────────────────────────────────────┐
│  OS Bootloader (GRUB, Windows BootMgr, etc.) │
│  - Loads the OS                              │
│  - Includes dm-crypt (Linux) or BitLocker-   │
│    equivalent (Windows) to read encrypted     │
│    sectors using the key handed off by PBA   │
└─────────────────────────────────────────────┘
```

## Authentication Modes

| Mode | Description | Trade-off |
|---|---|---|
| `Passphrase` | User must type a passphrase every boot. The PBA derives the volume key via Argon2id. | Most secure; ~50 ms Argon2id cost; defeats all boot-chain attacks. |
| `TpmOnly` | PBA asks the TPM to unseal the volume key. PCRs 0, 2, 4, 7 must match the recorded "known-good" state. | No typing required; vulnerable to evil-maid attack (boot, get key, install backdoor). |
| `TpmAndPassphrase` | Both: TPM PCRs must match AND user must type. | Recommended. Defeats evil-maid AND most offline attacks. |

## PCR Policy

| PCR | Measured by | Default selection |
|---|---|---|
| 0   | Firmware code (BIOS/UEFI) | ✓ |
| 1   | Firmware configuration |   |
| 2   | Option ROMs | ✓ |
| 3   | Option ROM configuration |   |
| 4   | MBR / bootloader | ✓ |
| 5   | Bootloader configuration |   |
| 6   | Host platform authorization |   |
| 7   | Secure Boot state | ✓ |
| 8-15 | OS-loaded (kernel, initrd) |   |

The default `{0, 2, 4, 7}` covers "is the boot chain trusted?".
PCR 5 is reserved for PBA's own measurement; PCR 8-15 are reserved
for the OS measured boot chain.

## PBA Configuration File

The PBA reads `pba.toml` from the EFI System Partition at boot. The
file is generated by `soteriad fde pba-config --out /boot/efi/EFI/Soteria/pba.toml`.

```toml
os_volume = "/dev/sda2"
kdf_m_cost_kib = 65536
kdf_t_cost = 3
kdf_p = 1
auth_mode = "TpmAndPassphrase"
pcrs = [0, 2, 4, 7]
locale = "en-US"
max_failed_attempts = 10
banner = "SOTERIA PRE-BOOT AUTHENTICATION\nAuthorized access only. All access is logged."
chain_load = "/EFI/systemd/systemd-bootx64.efi"
```

## Anti-Evil-Maid Defense

The TPM seal alone is necessary but not sufficient. The full
defense-in-depth:

1. **Secure Boot**: The UEFI firmware checks the PBA's signature
   against the platform key. A modified PBA is rejected.
2. **TPM PCR 4 sealing**: The PBA's hash is measured into PCR 4
   during boot. The sealed key is released only if PCR 4 matches
   the recorded value (which is the hash of the legitimate PBA).
3. **Recorded PCR digest**: The user records the PCR values at
   initial setup. Any subsequent unseal that doesn't match is
   rejected.
4. **Full-disk encryption of the PBA partition**: The PBA itself
   lives on an encrypted partition. An attacker who replaces it
   cannot decrypt the OS volume.

## Key Handoff to the OS

The PBA must pass the volume key to the OS. There are three common
methods:

### Linux: kernel command line + initrd
The PBA injects the key into the kernel command line:
```
soteria.key=01abc... rd.luks.uuid=...
```
The `soteria.initrd` (a custom initrd) reads the key and sets up
`dm-crypt` before `systemd` or `init` runs.

### Windows: BitLocker-style BCD
The PBA stores the key in a TPM NV index. The Windows bootloader
reads the NV index and uses it to unlock the BitLocker volume.

### macOS: AppleKeyStore
The PBA stores the key in the AppleKeyStore (AKS). The macOS
kernel's `AppleHFSController` reads the key and unlocks the volume.

## How to Build (when the binary lands)

```bash
# 1. Clone and build the PBA binary
git clone https://github.com/soteria-fs/soteria-pba
cd soteria-pba
cargo build --target x86_64-unknown-uefi --release

# 2. Sign with the platform key (replace with your key)
sbsign --key /path/to/DB.key --cert /path/to/DB.crt \
       --output bootx64.efi.signed target/x86_64-unknown-uefi/release/soteria-pba.efi

# 3. Copy to the EFI System Partition
sudo cp bootx64.efi.signed /boot/efi/EFI/Soteria/bootx64.efi

# 4. Generate the PBA config
soteriad fde pba-config --out /boot/efi/EFI/Soteria/pba.toml

# 5. Reboot and authenticate
sudo reboot
```
